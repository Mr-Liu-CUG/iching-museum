# Claude Code 项目开发实战经验

> 基于「易经六十四卦 · 数字博物展厅」项目（~18,000 行 / 13 文件 / 4 语言 / 6 阶段）的完整开发过程总结。

---

## 一、核心原则

### Claude Code 最适合的场景

```
适合 ✅                              谨慎 ❌
───────────────────────────────    ───────────────────────────────
从清晰需求出发的多文件实现            需求本身还在探索中
有明确验收标准的任务                  需要实时视觉反馈的 UI 微调
可拆分为独立子任务的并行工作          依赖外部 API / 认证的操作
多轮迭代完善的复杂功能                状态极度依赖之前对话上下文
```

### 效率公式

> **效率 = 需求清晰度 × 任务拆分粒度 × 并行度 ÷ 返工次数**

本项目高效的核心原因：需求极具体（序厅文案、CSS 参数、i18n 键名全部预定义）→ 阶段交付物明确 → 大量并行执行 → 每步立即验证。

## 二、项目启动：先规划，后动手

### 何时必须进入 Plan Mode

| 场景 | 是否 Plan Mode |
|------|---------------|
| 新项目从零开始 | ✅ 必须 |
| 涉及 3+ 文件修改 | ✅ 必须 |
| 需要设计新的数据流/架构 | ✅ 必须 |
| 有多种技术方案可选 | ✅ 必须 |
| 单文件小修改 / 修 bug / 加一个函数 | ❌ 可跳过 |

### 好计划的 7 要素

1. **问题定义** — 当前状态 vs 目标状态，用具体指标
2. **架构图** — ASCII 流程图，一目了然
3. **文件清单** — 每个文件改什么、新建还是修改
4. **数据结构** — 新增的 API 签名或数据格式
5. **实施顺序** — 有依赖关系的步骤序列（Phase 1→2→3…）
6. **验证方式** — 每步如何确认"做对了"
7. **风险点** — 可能出错的地方及预案

### 本项目最关键的架构决策

在 Plan Mode 中我们确定了**双轨 i18n 架构**，避免了灾难性设计：

```
❌ 拒绝方案：把翻译塞进 hexagrams-data.js
   → 单个文件超 2MB，切换语言卡顿，维护噩梦

✅ 采用方案：UI 标签（I18n.t）+ 内容翻译（I18n.content）分离
   → 按需加载翻译文件，源语言零开销，优雅降级
```

**教训：架构决策必须在 Plan Mode 中完成，不要在写代码时临时决定架构。否则返工成本极高。**

## 三、需求表达：越具体，越精准

### 需求颗粒度对比

**模糊需求（产出模糊结果）：**
> "给网页加个介绍易经历史的部分"

**具体需求（本项目实际采用）：**
> "在 header 和 main 之间加序厅区域，四个 tab 面板：
> 1. 易道渊源 — CSS Timeline，4 个历史阶段，每阶段标题+正文
> 2. 揲蓍求卦 — 2 张方法卡片，大衍筮法四步流程+金钱卦四条规则
> 3. 历代圣贤 — 4 张大师卡片，姓名+学派+简介
> 4. 现代启示 — 3 张卡片，二进制/共时性/危机管理
> 样式：宋韵美学、1px 淡金 Timeline 轴线、❖ 节点、行高 1.8-2.0、字间距 0.05em
> i18n：所有文本 data-i18n 属性，四语包补齐对应 key"

### 实操技巧

| 技巧 | 示例 |
|------|------|
| 给出具体文案 | 不写"介绍伏羲"，直接粘贴完整段落 |
| 给出 CSS 精确值 | 不写"好看的线"，写 `1px solid rgba(176,141,87,0.25)` |
| 给出一致性约束 | "四个语言包的 key 完全一致，zh-CN.js 是源" |
| 给出验收标准 | "切换语言后序厅无残留中文" |
| 用 ASCII 图表达布局 | 比文字描述快 10 倍被准确理解 |

### 让 Claude Code 帮你细化需求

如果你只有模糊想法：
> "我想做一个易经介绍页面，包含历史、起卦方法、重要人物和现代应用。请帮我细化每个板块应该具体包含什么内容、用什么布局。"

Claude Code 会给出几种方案，你选择后再进入实现。

## 四、并行策略：能并行就不串行

### 本项目并行实践

```
阶段               任务                              策略
─────────────────────────────────────────────────────────
语言包翻译    简中(源) → 用户自己                  ┐
             英文      → Agent A                  │ 4 个 Agent
             繁中      → Agent B                  │ 同时执行
             日文      → Agent C                  ┘

内容翻译      content-en.js (978行)               ┐ 2 个 Agent
             content-ja.js (359行)               ┘ 后台并行

JS 验证      node --check × 9 个文件              → 一键批量
```

**效率差距：串行 12 分钟 → 并行 3 分钟。累计节省约 60% 时间。**

### 判断标准

> **两个任务是否共享任何产出物？如果是 → 串行。如果否 → 并行。**

```
✅ 可并行：
Agent A: "翻译 en.js 的 57 个序厅 key"
Agent B: "翻译 ja.js 的 57 个序厅 key"
→ 两个文件互不依赖

❌ 必须串行：
Agent A: "创建 CSS 样式"
Agent B: "写 JS 逻辑调用这些样式"
→ B 需要知道 A 的 class 名称
```

### Agent 任务描述要自包含

```markdown
❌ 太模糊：
"翻译日语语言包"

✅ 自包含：
"读取 data/lang/zh-CN.js 中的 57 个序厅 key（第120-197行），
创建 data/lang/ja.js，保持 key 完全一致，将值翻译为日语。
注意：使用自然日语，非机器翻译风格。完成后用 node --check 验证语法。"
```

一个好的 Agent 任务描述包含：**源文件路径、具体范围（行号）、输出路径、约束条件、验收方式。**

## 五、分阶段交付：小块验证，大块组装

### 本项目的 6 阶段拆分

```
Phase 1: HTML 序厅结构 (index.html +180行)
   → 验收：在浏览器中看到四个 tab 面板（无样式、无交互）

Phase 2: CSS 序厅样式 (style.css +200行)
   → 验收：序厅在视觉上呈现宋韵风格（Timeline 线条、卡片布局）

Phase 3: JS 序厅逻辑 (script.js +59行)
   → 验收：展开/折叠、Tab 切换交互正常

Phase 4: 四语包更新 (4个文件 × +57键)
   → 验收：每个语言包 node --check 通过

Phase 5: 内容翻译扩展 (content-ja.js 54→359行)
   → 验收：所有 64 卦在日语版有现代解释

Phase 6: 全项目验证
   → 验收：9 个 JS 文件语法检查通过 + 浏览器端到端测试
```

### 阶段拆分的黄金法则

1. **每阶段 1-2 个文件**，不超过 200 行变更
2. **每阶段有独立的验收标准**，不依赖后续阶段
3. **每个阶段完成后立即验证**，不在错误基础上继续建造
4. **HTML → CSS → JS 按顺序**，这是天然的最佳依赖链

## 六、上下文管理：跨会话开发的命脉

### 本项目经历了上下文压缩（/compact）

当对话上下文达到上限时，Claude Code 会自动生成摘要。为了确保压缩后能无缝继续：

| 策略 | 本项目做法 |
|------|-----------|
| **Plan 文件持久化** | 计划写入 `.claude/plans/` 目录，压缩后自动加载 |
| **关键决策记录** | 架构选型、数据索引方式在计划中详细说明 |
| **完成状态标记** | 用 ✅ 明确标记每个 Phase 的完成状态 |
| **文件行数记录** | 每个文件的精确行数便于快速定位 |
| **待办事项明确** | Pending Tasks 列表中标记未完成项 |

### 上下文压缩后的恢复技巧

压缩后续接时，Plan 文件会被自动注入 system-reminder。此时：
1. **先看 Plan 中的完成状态** — 确认哪些 Phase 已完成
2. **验证关键文件存在** — 用 `find` / `ls` 确认新建文件已落地
3. **做一次快速语法检查** — 确保上次会话的代码无遗漏错误
4. **从第一个未完成 Phase 开始** — 不要重复已完成工作

## 七、验证机制：每一步都可回溯

### 本项目的验证链

```bash
# 每次修改 JS 文件后立即验证（本项目累计执行 20+ 次）
find . -name "*.js" -not -path "*/node_modules/*" -exec node --check {} \;

# 确认文件行数在预期范围
wc -l index.html style.css script.js data/lang/*.js

# 确认新建文件存在
ls -la data/content-en.js data/content-ja.js data/content-zh-TW.js

# 确认关键 HTML 元素存在
grep -c 'data-i18n' index.html
```

### 验证的时间点

```
每完成一个文件      → 语法检查
每完成一个 Phase    → 功能验证 + 回归检查
交付前              → 全项目语法 + 文件完整性 + 浏览器测试
```

**原则：不要在错误的基础上继续建造。发现错误立即修复，不累积。**

## 八、错误处理：快速定位，精准修复

### 本项目遇到的实际问题

**问题 1：`grep -P` 在 Windows bash 中失败**
```
grep -P "pattern" → "supports only unibyte and UTF-8 locales"
原因：Windows Git Bash 的 grep 不支持 -P（Perl 正则）
解决：使用 Claude Code 内置的 Grep 工具（基于 ripgrep，跨平台一致）
```

**经验：优先使用 Claude Code 的专用工具（Grep/Glob/Read）而非 shell 命令，它们跨平台、结果结构化、自动分页。**

**问题 2：Edit 替换时误删相邻代码**
```
原因：old_string 匹配范围不精确，覆盖了相邻的注释行
解决：扩大 old_string 匹配范围，包含更多上下文使其唯一
```

**经验：使用 Edit 工具时，old_string 应该包含足够的上下文行（至少前后各 1-2 行）以保证唯一性。**

## 九、多语言项目：先锚定源语言

### 本项目 i18n 流程

```
Step 1: 确定源语言（zh-CN.js）
        → 先完成所有 key 的中文值
        → 每个 key 命名遵循规范：{模块}.{子模块}.{字段}

Step 2: 源语言作为翻译模板
        → Agent 直接读取 zh-CN.js 翻译为其他语言
        → 不需要重复描述每个 key 的含义

Step 3: 内容翻译独立管理
        → UI 标签（110 键）和内容翻译（64 卦）分开
        → 内容翻译按语言拆分文件，按需加载

Step 4: 繁体使用 S→T 映射表
        → 不存储完整繁体副本
        → 运行时字符转换，节省 90% 繁体维护成本
```

### Key 命名规范

```
✅ 好的命名：
"prologue.tab.history"      ← 模块.子模块.字段
"method.coin1.label"         ← 层级清晰
"masters.name1"              ← 编号统一

❌ 差的命名：
"tab1"                       ← 不知道是什么
"history_era1_title"         ← 用下划线混用
"methodIntro"                ← 驼峰与点号混用
```

## 十、前端项目：静态优先，渐进增强

### file:// 协议兼容

本项目核心约束：必须支持双击 HTML 直接运行，不能用 `fetch()`。

```
✅ 可用技术                    ❌ 不可用技术
<script src="...">            fetch()
<link rel="stylesheet">       import() 动态模块
内联所有逻辑                  任何需要 HTTP 服务器的方案
```

### 依赖加载顺序

```
index.html 中的 <script> 顺序：

1. data/lang/zh-CN.js    ← 默认语言包（同步加载，避免 UI 闪烁）
2. data/hexagrams-data.js ← 数据层（defer）
3. script.js              ← 控制层（defer）

其他语言包和内容翻译 → 运行时按需动态注入 <script> 标签
```

**原则：默认语言包同步加载 → 首屏无闪烁；其他语言按需加载 → 不浪费带宽；所有数据/逻辑用 defer → 不阻塞渲染。**

## 十一、大型数据文件：用脚本生成，不手写

### hexagrams-data.js 的生成方式

```
源数据：3 个 CSV 文件（爻辞解释/入门数据/记忆口诀）
       + zhouyi-main/docs 目录（Markdown 格式）

构建脚本：scripts/build-data.js (676 行)
       → 读取 CSV + Markdown
       → 合并、去重、数据结构化
       → 输出 hexagrams-data.js (12,625 行)

流程：
修改 CSV → 运行 node scripts/build-data.js → 生成 JS → 验证
```

**教训：12,000 行的数据文件绝不手写。写一个构建脚本，修改数据源 → 一键生成。这比手动编辑快 100 倍且零错误。**

## 十二、常见陷阱与避坑指南

### 陷阱 1：跳过 Plan 直接写代码

```
症状：写了 500 行后发现架构不对，全部推倒重来
预防：超过 2 个文件变更的任务 → 先 Plan Mode
本项目：Plan 阶段的架构决策（双轨 i18n）避免了至少 3 次大规模返工
```

### 陷阱 2：一次性交付太多

```
症状：攒了 5 个 Phase 一起提交，中间某个 Phase 出错全部回滚
预防：每完成一个 Phase 立即验证 → 确认正确 → 再开始下一个
本项目：6 个 Phase 逐个交付，每个完成后验证
```

### 陷阱 3：Agent 任务描述模糊

```
症状：Agent 返回的结果不是你想要的
预防：在任务描述中包含：
  - 具体要读的文件路径（含行号范围）
  - 输出文件路径和格式要求
  - 约束条件（"使用自然日语，非机器翻译"）
  - 验收标准（"完成后用 node --check 验证语法"）
```

### 陷阱 4：忽略文件现状就开始改

```
症状：基于"记忆中的代码"修改，实际文件已变化
预防：修改前先 Read 目标文件，确认当前状态
本项目：每次修改前都先 Read，避免基于过时记忆操作
```

### 陷阱 5：不验证就认为完成

```
症状：写了 500 行代码，觉得没问题，结果一跑全是语法错误
预防：JS → node --check，CSS → 检查关键选择器，HTML → 检查关键元素
本项目：累计执行 node --check 验证 20+ 次
```

## 总结：高效使用 Claude Code 的 10 条定律

| # | 定律 | 一句话 |
|---|------|--------|
| 1 | **Plan First** | 多文件任务坚决先进 Plan Mode |
| 2 | **需求为王** | 需求越具体，产出越精准 |
| 3 | **化整为零** | 每阶段 1-2 文件，独立可验证 |
| 4 | **能并则并** | 互不依赖的任务一律并行 |
| 5 | **即改即验** | 每改完一个文件立即验证语法 |
| 6 | **就近读文件** | 修改前先 Read 确认当前状态 |
| 7 | **用专用工具** | Grep/Glob 优于 shell 命令 |
| 8 | **Agent 自包含** | 任务描述 = 路径 + 格式 + 约束 + 验收 |
| 9 | **约定优于配置** | 统一的命名规范比文档更有效 |
| 10 | **数据脚本化** | 大文件用脚本生成，绝不手写 |
